# Bug 診斷必須走完五步驟再告知

## 問題

Claude 看到 bug 報告，直接跳去改程式碼，沒有先確認根因。改了之後問題依然存在，因為一開始的假設就是錯的。或者，Claude 改了一半告訴用戶「我懷疑是 X，你覺得呢？」——在還沒確定之前就開始問。

## 原因

Claude 的訓練讓它傾向快速給出回答，「先動手再說」比「先確認再動手」更符合訓練時的回饋模式。但 debug 的本質是推理，跳過推理直接修通常只是在猜。

## 解決方案

Bug 修復流程必須按照固定順序：

1. **蒐集線索**：讀 error log、讀相關程式碼、確認環境狀態
2. **列出可能原因**：把所有可能的根因列出來（不是只選一個）
3. **用工具逐一排除**：每個假設都要有對應的診斷指令來驗證或排除
4. **確定根因**：有足夠證據指向單一根因才繼續
5. **一次修復 + 自驗 + 才告知**：改完之後自己驗證（跑 test、curl 確認等），確認有效才向用戶回報

不到步驟 5 不說話。

## 可複用的 CLAUDE.md 規則

```markdown
## Bug 修復 SOP
- 順序：蒐集線索 → 列出可能原因 → 工具逐一排除 → 確定根因 → 一次修復 → 自驗 → 才告知
- 未到「確定根因」前禁止動手改程式
- 禁止「我猜可能是 X，你覺得呢？」——猜測要自己排除，不問用戶
- 修復後必須自驗（跑測試/curl 確認）才能回報「修好了」
```
